In the previous lesson, we explored Koan Error Handling. Now we are going to take a look at Generic Functions, which is an important concept in Krypton. As you may already be aware, many programming languages have some way of talking about functions that work over many types instead of being tied to one concrete type, and Krypton is no exception to that general pattern. It is worth mentioning, however, that the exact spelling still matters, because small syntax choices tend to shape how comfortably a reader can move through later code.

The example below demonstrates that type parameters appear in square brackets after the function name, as in `fun identity[a](x: a) -> a = x`. Note that the tiny identity example is famous for a reason, because it strips away everything except the idea that the same function shape can be reused across many concrete types. This may sound somewhat ordinary, and in a sense it is, but that is also why it is useful to see it written out plainly before the lessons get more layered. Interestingly, this is somewhat similar to parametric polymorphism in typed languages, although the Krypton version has its own punctuation and naming choices that should be read carefully.

With that in mind, let us now turn our attention to what the program is doing at runtime and at compile time. It is generally considered best practice to make one small change and see what changes elsewhere, even if the change is something as minor as a different literal, branch, field name, or type annotation.
